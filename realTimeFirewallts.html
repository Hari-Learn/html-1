🔥 Real-Time Firewall Troubleshooting Guide

1. Internet Not Working Behind Firewall – What to Check?

  • Check physical connectivity (interface status, link up/down).
  • Verify IP address, subnet mask, and default gateway on client.
  • Ping firewall interface from client.
  • Check DNS configuration.
  • Verify outbound policy allows traffic.
  • Check NAT configuration.
  • Test connectivity from firewall (ping/traceroute).

2. How to Identify NAT, Policy, or Routing Issue?

  • NAT Issue: Traffic leaving but no response. Check SNAT/DNAT rules.
  • Policy Issue: Traffic blocked immediately. Check firewall rules and hit counts.
  • Routing Issue: Traffic not reaching destination. Verify routing table and next hop.
  • Use packet capture or debug tools.

3. Policy Configured but Traffic Blocked – Why?

  • Rule order – another rule may be overriding.
  • Incorrect source/destination zones or IPs.
  • Service/port mismatch.
  • Application control or security profiles blocking traffic.
  • Schedule restriction on policy.
  • Implicit deny rule at bottom.

4. How to Use Logs for Troubleshooting?

  • Filter logs by source/destination IP.
  • Check action (allow/deny).
  • Identify which policy is matched.
  • Verify NAT translation details.
  • Check timestamps and session states.
  • Use real-time logs or CLI debug.

5. Interface Issue vs Policy Issue – How to Verify?

  • Interface Issue:
    • Interface down or no link.
    • No IP assigned.
    • No ARP resolution.
  • Policy Issue:
    • Traffic hits firewall but gets denied.
    • No matching rule or wrong rule.
  • Use ping, traceroute, and logs to differentiate.

6. Real-Time Troubleshooting Tools

  • Ping – test reachability.
  • Traceroute – identify path and hops.
  • Packet Capture – analyze live traffic.
  • Session Table – check active sessions.
  • Debug Flow / CLI Debug – trace packet processing.
  • Real-time logs / monitoring dashboard.

7. VPN Not Connecting – Troubleshooting Steps

  • Check internet connectivity on both ends.
  • Verify VPN configuration (IP, pre-shared key, certificates).
  • Ensure Phase 1 (IKE) and Phase 2 settings match.
  • Check firewall policies allowing VPN traffic.
  • Verify NAT traversal settings.
  • Review VPN logs for errors.
  • Check routing for remote network.

8. Why Policy Order Matters

Firewalls process rules top-down. The first matching rule is applied.

  • If a deny rule is placed above an allow rule, traffic will be blocked.
  • Incorrect ordering can block valid traffic.
  • Always place specific rules above general rules.
  • Review rule hit counts to optimize order.
