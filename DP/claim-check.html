Claim Check Pattern

A messaging pattern that stores large payloads externally and sends only a lightweight reference (claim check) through the messaging system.

The Problem

Message brokers work best with small messages. Large files, documents, images, or reports can:

  • Increase network traffic
  • Slow down message processing
  • Consume broker storage
  • Reduce overall system performance

Example: Sending a 100 MB PDF through a message queue.

The Solution

Store the large payload in external storage and send only a unique identifier or reference in the message.

Large File
→
External Storage
→
File ID
→
Message Queue

Example Message 

Instead of:

{
  "file": "100MB PDF content..."
}

Send:

{
  "documentId": "DOC-12345"
}

The consumer retrieves the actual document using the provided identifier.

Workflow

  1. 1. Store large payload in external storage.
  2. 2. Generate a unique claim check ID.
  3. 3. Send only the ID through the message queue.
  4. 4. Consumer receives the message.
  5. 5. Consumer retrieves the payload using the ID.
  6. 6. Process the payload.

Common Use Cases

📄 Document Processing

Large PDFs and reports stored outside the queue.

🖼️ Image Processing

Send image references instead of image binaries.

🎥 Media Pipelines

Store videos in object storage and send IDs.

☁️ Cloud Applications

Common with S3, Azure Blob Storage, and GCS.

Advantages

  • ✔ Smaller messages
  • ✔ Faster message processing
  • ✔ Reduced broker storage usage
  • ✔ Better scalability
  • ✔ Efficient handling of large files

Challenges

  • ✖ Additional storage dependency
  • ✖ Extra retrieval step
  • ✖ Storage cleanup required
  • ✖ Handling missing references
  • ✖ More complex architecture

Key Takeaway

The Claim Check Pattern improves messaging efficiency by replacing large payloads with lightweight references. It is especially valuable in distributed systems that process documents, media files, and other large data objects.
