Null Object Design Pattern

Replaces null references with a special object that provides default or "do nothing" behavior.

The Problem

Applications often contain many null checks to avoid runtime errors such as NullPointerException. 

if(customer != null) {
    customer.sendNotification();
}

As the application grows, these checks become repetitive and clutter the code.

The Solution

Instead of returning null, return a special object that implements the same interface but performs default behavior.

→

Example 

interface Customer {
    void sendNotification();
}

class RealCustomer implements Customer {
    public void sendNotification() {
        System.out.println("Notification sent");
    }
}

class NullCustomer implements Customer {
    public void sendNotification() {
        // Do nothing
    }
}

Before vs After

Before 

if(customer != null) {
    customer.sendNotification();
}

After 

customer.sendNotification();

Common Use Cases

  • 👤 Guest or anonymous users
  • 📢 Notification systems
  • 📝 Logging frameworks
  • 🎮 Game AI fallback behaviors
  • 🔌 Optional services or plugins

Advantages

  • ✔ Eliminates repeated null checks
  • ✔ Cleaner and more readable code
  • ✔ Reduces NullPointerExceptions
  • ✔ Simplifies client logic

Challenges

  • ✖ Extra classes to maintain
  • ✖ Can hide missing data issues
  • ✖ May make debugging harder
  • ✖ Not suitable for every scenario

Key Takeaway

The Null Object Pattern avoids special handling for null values by providing a harmless default implementation. It helps simplify client code and makes applications easier to maintain.
