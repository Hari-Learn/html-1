Type Object Design Pattern

Represents "types" as objects instead of creating a class for every variation. This allows new types to be added dynamically without changing code.

The Problem

Imagine a game with many monster types:

  • Dragon
  • Goblin
  • Orc
  • Skeleton
  • Zombie

Creating a separate class for every type can make the codebase difficult to maintain and extend.

The Solution

Store type information in separate objects and let entities reference them.

Monster
→
Type Object
→
Dragon / Goblin / Orc

Example 

MonsterType
------------
name = "Dragon"
health = 500
attack = 80

Monster
------------
name = "Fire Dragon"
type = Dragon

Multiple monsters can share the same type definition, reducing duplication.

Structure

Type Object

Stores shared properties such as health, attack, speed, and abilities.

Instance Object

Stores unique runtime data such as current health and position.

Common Use Cases

  • 🎮 Game development (monsters, weapons, characters)
  • 🛒 Product catalogs
  • 🚗 Vehicle configuration systems
  • 🏢 Enterprise metadata systems
  • 📦 Inventory management

Advantages

  • ✔ Add new types without code changes
  • ✔ Reduces class explosion
  • ✔ Easier configuration management
  • ✔ Better data-driven design

Challenges

  • ✖ More indirect object relationships
  • ✖ Runtime validation required
  • ✖ Can be harder to debug
  • ✖ Less compile-time safety
