Monostate (Borg) Design Pattern

Multiple object instances share the same state, making them behave as if they were a single object.

What is Monostate?

The Monostate (also called the Borg Pattern) is an alternative to the Singleton pattern. Instead of restricting object creation to one instance, it allows multiple instances while sharing the same internal state across all objects.

Singleton vs Monostate

Singleton

  • Only one instance exists.
  • Global access point.
  • Harder to test.

Monostate

  • Multiple instances allowed.
  • All instances share state.
  • More flexible design.

Example 

obj1 = Borg()
obj2 = Borg()

obj1.name = "Alice"

print(obj2.name)
# Output: Alice

Even though obj1 and obj2 are different objects, they share the same state.

Common Use Cases

Configuration

Share application settings across objects.

Caching

Maintain a common cache state.

Shared Context

Keep synchronized data across instances.

Advantages

  • Allows multiple object instances.
  • Reduces Singleton restrictions.
  • Easier to extend and test.

Disadvantages

  • Shared state can cause side effects.
  • Harder to track data changes.
  • Not thread-safe by default.
