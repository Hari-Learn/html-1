Outbox Pattern

A reliability pattern that ensures database updates and event/message publishing happen consistently without losing messages during failures.

The Problem

Consider an order service that saves an order and then publishes an event. 

1. Save Order to Database
2. Publish OrderCreated Event

What happens if the database update succeeds but the event publication fails?

  • Order exists in the database.
  • No event is published.
  • Other services never know the order was created.

The Solution

Store both the business data and the event in the same database transaction. A background process later publishes the event safely.

Save Order
→
Write Outbox Record
→
Publish Event Later

Example Database Tables

Orders Table 

id | customer
----------------
101 | John

Outbox Table 

id | event
------------------------
1  | OrderCreated

Workflow

  1. 1. Begin database transaction.
  2. 2. Save business data (Order).
  3. 3. Insert event into Outbox table.
  4. 4. Commit transaction.
  5. 5. Background worker reads Outbox records.
  6. 6. Publish events to message broker.
  7. 7. Mark Outbox record as processed.

Advantages

  • ✔ Prevents lost events
  • ✔ Reliable message delivery
  • ✔ No distributed transactions needed
  • ✔ Works well with microservices
  • ✔ Improves fault tolerance

Challenges

  • ✖ Additional Outbox table
  • ✖ Background worker required
  • ✖ Event cleanup needed
  • ✖ Increased operational complexity
  • ✖ Duplicate events may occur

Common Use Cases

🛒 E-Commerce

Publish order and inventory events reliably.

💳 Payment Systems

Ensure payment events are never lost.

📦 Logistics

Track shipment updates across services.

☁️ Event-Driven Systems

Reliable integration between microservices.

Key Takeaway

The Outbox Pattern guarantees that database changes and published events stay consistent by storing events in an Outbox table as part of the same transaction. It is one of the most widely used reliability patterns in modern microservices architectures.
