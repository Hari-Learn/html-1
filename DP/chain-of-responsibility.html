Chain of Responsibility

Behavioral Design Pattern

The Chain of Responsibility pattern allows a request to travel through a chain of handlers. Each handler decides whether it can process the request or pass it to the next handler.

Request Flow

📩 Request
→
👨‍💻 Handler 1
→
⚙️ Handler 2
→
👨‍💼 Handler 3
🔄

Decoupling

Sender doesn't know which handler will process the request.

⚡

Flexibility

Handlers can be added, removed, or reordered dynamically.

🧩

Reusability

Individual handlers can be reused in different chains.

Customer Support Example

🟢 Level 1 Support

Handles password resets, account questions, and basic issues.

🟡 Level 2 Support

Handles technical bugs, system failures, and configuration issues.

🔴 Manager

Handles critical complaints and business-impacting problems.

💡 If Level 1 cannot resolve the issue, it forwards the request to Level 2. If Level 2 still cannot handle it, the Manager receives it.

✅ Advantages

  • ✔ Reduces coupling between sender and receiver
  • ✔ Easy to add new handlers
  • ✔ Improves maintainability
  • ✔ Supports dynamic request processing

❌ Disadvantages

  • ✖ Request may go unhandled
  • ✖ Debugging can be difficult
  • ✖ Long chains may affect performance
  • ✖ Complex handler management

Chain of Responsibility
