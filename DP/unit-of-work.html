Unit of Work

A design pattern that maintains a list of objects affected by a business transaction and coordinates writing changes to the database as a single operation.

What is Unit of Work?

The Unit of Work pattern tracks changes made during a business transaction and ensures that all updates are committed together. This improves consistency, reduces database round trips, and helps maintain transactional integrity.

Core Responsibilities

📋

Track Changes

Monitors newly created, updated, and deleted entities during a transaction.

💾

Commit Data

Saves all tracked changes to the database in a single coordinated operation.

🔄

Transaction Control

Ensures all operations succeed or fail together using transactional boundaries.

🛡️

Data Integrity

Prevents inconsistent states by ensuring reliable and atomic updates.

Unit of Work Lifecycle

1️⃣

Start

Begin a transaction and create a Unit of Work instance.

2️⃣

Track

Register entities that are added, modified, or removed.

3️⃣

Validate

Apply business rules and ensure consistency.

4️⃣

Commit

Persist all changes in a single transaction.

Benefits of Unit of Work

  • ✅ Reduces database calls
  • ✅ Improves transaction management
  • ✅ Centralizes persistence logic
  • ✅ Ensures data consistency
  • ✅ Simplifies repository coordination
  • ✅ Supports rollback scenarios
