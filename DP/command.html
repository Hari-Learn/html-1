Command Pattern

Behavioral Design Pattern

The Command Pattern encapsulates a request as an object, allowing requests to be parameterized, queued, logged, and even undone. It separates the object that invokes an operation from the one that actually performs it.

Command Flow

👤 Client
→
📦 Command
→
🎮 Invoker
→
⚙️ Receiver
👤

Client

Creates command objects and assigns receivers.

📦

Command

Encapsulates a request into an executable object.

🎮

Invoker

Executes commands without knowing implementation details.

⚙️

Receiver

Performs the actual business operation.

TV Remote Control Example

📺 Turn TV On

Executes an OnCommand object.

🔇 Turn TV Off

Executes an OffCommand object.

🔊 Increase Volume

Executes a VolumeUpCommand object.

💡 Every button on the remote is represented as a separate command object. The remote only triggers commands and does not know how the TV works internally.

Real-World Use Cases

↩️

Undo/Redo

Text editors and design tools.

📋

Task Queues

Execute jobs asynchronously.

🎮

Game Controls

Map actions to buttons.

🏠

Smart Homes

Turn devices on/off remotely.

✅ Advantages

  • ✔ Decouples sender and receiver
  • ✔ Supports undo and redo
  • ✔ Easy command logging
  • ✔ Supports queuing operations
  • ✔ Extensible architecture

❌ Disadvantages

  • ✖ Many command classes required
  • ✖ Increased code complexity
  • ✖ More objects to manage
  • ✖ Can be overkill for simple tasks
