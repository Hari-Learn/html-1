Transactional Outbox Pattern

The Transactional Outbox pattern ensures that database updates and event/message publishing happen reliably together, without losing events.

Instead of publishing a message directly after updating the database, the application stores the message in an Outbox Table within the same database transaction.

How It Works

  1. Update business data.
  2. Insert an event into the Outbox table.
  3. Commit both actions in a single transaction.
  4. A background process reads the Outbox table.
  5. The event is published to a message broker.
  6. The Outbox record is marked as processed.

Benefits

  • Prevents lost messages.
  • Maintains data consistency.
  • Works well with microservices.
  • Avoids distributed transactions.

Simple Flow

Application
↓
Database Transaction
├── Business Data
└── Outbox Event
↓
Outbox Processor
↓
Message Broker
