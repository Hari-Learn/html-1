Data Mapper Pattern

An integration pattern that separates business objects from database operations by using a dedicated mapper layer to transfer data between the application and the database.

Overview

The Data Mapper pattern keeps domain objects independent of the database. Instead of embedding SQL or persistence logic inside business entities, a mapper component handles all data access and object-relational mapping. This separation improves maintainability, testability, and flexibility.

Key Components

Domain Object

Represents business data and rules without database knowledge.

Data Mapper

Translates data between domain objects and database records.

Database

Stores and retrieves persistent application data.

Data Flow

Domain Object
⇄
Data Mapper
⇄
Database

Example Scenario

A User object exists in the application:

  • User ID: 101
  • Name: John Doe
  • Email: john@example.com

The Data Mapper converts the User object into database records and reconstructs the object when data is retrieved.

Benefits

  • Separates business logic from persistence logic.
  • Improves maintainability and code organization.
  • Makes unit testing easier.
  • Supports multiple database implementations.
  • Works well in Domain-Driven Design (DDD) architectures.
