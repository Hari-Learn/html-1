Envelope-Letter Idiom

A C++ design technique used to hide implementation behind a wrapper.

What is the Envelope-Letter Idiom?

The Envelope-Letter Idiom separates an object's interface from its implementation. The public object acts as an Envelope, while the actual implementation is the Letter hidden inside.

Goal: Hide implementation details while exposing a clean public interface.

How It Works

📩 Envelope

The wrapper class that users interact with. It forwards requests to the hidden implementation.

📄 Letter

The actual implementation class that performs the work. Users never access it directly.

Structure

Application
      │
      ▼
+-------------------+
|   Envelope Class  |
|  (Public Wrapper) |
+-------------------+
          │
          ▼
+-------------------+
|   Letter Class    |
| Hidden Implementation |
+-------------------+

Advantages

✅ Information Hiding

Implementation details remain private.

✅ Cleaner Interface

Users interact with a simple public API.

✅ Easier Maintenance

Implementation changes rarely affect client code.

✅ Better Flexibility

Implementation can be replaced without changing the interface.

Disadvantages

  • Extra level of indirection.
  • Additional dynamic memory usage in many implementations.
  • Slight runtime overhead due to forwarding calls.
  • Requires more boilerplate code.

Real-World Example

Many GUI frameworks and large C++ libraries use wrapper classes to hide complex rendering engines, networking code, or platform- specific implementations from application developers.

ELI10 (Explain Like I'm 10)

Imagine your friend sends you a birthday card.

  • 📩 The Envelope protects what's inside.
  • 📄 The Letter contains the actual message.
  • You open the envelope only when you need to read the letter.
In C++:
The Envelope is the public wrapper class.
The Letter is the hidden implementation.

Users interact with the wrapper, while the real work happens inside the hidden implementation.
