Fallback Pattern

What is it?

The Fallback Pattern is a resilience design pattern used in microservices to provide an alternative response when a service fails or becomes unavailable. Instead of showing an error, the system gracefully returns backup data, a default response, or a cached value.

ELI10 Explanation

It’s like carrying an umbrella ☔. If it rains (failure happens), you don’t get wet — you already have a backup plan ready!

Why is it Important?

  • Prevents bad user experience (no ugly error pages)
  • Keeps applications partially functional
  • Improves system reliability and availability
  • Handles temporary failures gracefully

Types of Fallbacks

  • Static Fallback: Show a fixed message (e.g., "Service unavailable")
  • Cached Data: Return last known data
  • Default Value: Return safe default (e.g., empty list)
  • Alternative Service: Call another service as backup

Real-World Examples

  • Weather app shows yesterday’s data if API fails 🌦️
  • E-commerce site shows "popular products" if recommendations fail 🛒
  • Video platform shows thumbnails even if video fails to load 🎥

Simple Flow

Request → Service Call → ❌ Failure → Fallback Triggered → ✅ Backup Response

Code Example (Pseudo)

function getWeather() {
  try {
    return callWeatherAPI();
  } catch (error) {
    return "Showing cached weather data 🌤️";
  }
}

When to Use?

  • When services depend on external APIs
  • In distributed systems with network failures
  • For non-critical features (recommendations, analytics)

When NOT to Use?

  • Critical systems like payments 💳 (wrong data is dangerous)
  • When fallback may mislead users
