API Gateway & BFF Patterns

Understanding modern backend architecture patterns

Explain Like I'm 10 👶

Imagine a restaurant 🍔:

👉 API Gateway = Waiter who takes your order and sends it to the kitchen 👉 BFF = Special chef who prepares food exactly how YOU like it

The waiter handles everyone, but the chef customizes your meal.

🌐

API Gateway

A single entry point that handles all client requests and routes them to the correct backend services.

How It Works

  • Client sends request
  • Gateway validates & authenticates
  • Routes request to correct service
  • Returns response to client

Responsibilities

  • Authentication & authorization
  • Rate limiting
  • Caching responses
  • Logging & monitoring
📱

Backend for Frontend (BFF)

A backend tailored for a specific frontend (web, mobile, etc.), providing exactly the data it needs.

How It Works

  • Frontend calls its dedicated BFF
  • BFF fetches data from multiple services
  • Combines & formats response
  • Returns optimized data

Benefits

  • No over-fetching or under-fetching
  • Faster frontend performance
  • Custom logic per device
  • Simpler frontend code

When to Use API Gateway

  • Microservices architecture
  • Need centralized control
  • Security & monitoring required

When to Use BFF

  • Different frontends (web vs mobile)
  • Need customized responses
  • Performance optimization

Key Difference

API Gateway manages and routes requests.

BFF customizes responses for specific frontends.

👉 Gateway = Control & routing layer 👉 BFF = Customization layer

Typical Architecture Flow

Client → API Gateway → BFF → Microservices → Database
