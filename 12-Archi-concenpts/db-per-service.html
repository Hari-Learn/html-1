Database per Service Pattern

The Database per Service Pattern ensures that each microservice owns and manages its own database. This prevents tight coupling between services and allows each service to evolve independently.

Instead of sharing a single database, every service has its own private data store. Other services can only access this data through APIs, not direct database queries.

🧒 Explain Like I'm 10

Imagine a school where each teacher keeps their own notebook:

  • Math teacher has a math notebook 📘
  • Science teacher has a science notebook 📗
  • English teacher has an English notebook 📙

If you want math marks, you ask the math teacher — you don’t peek into their notebook. This way, everyone can work independently without messing up each other's notes!

✔ Benefits

  • Loose coupling between services
  • Independent scaling and deployment
  • Better performance (optimized per service)
  • Freedom to choose different database types

⚠ Challenges

  • Data duplication across services
  • Complex distributed queries
  • Requires eventual consistency
  • Cross-service transactions are harder

How It Works

  1. User sends request (e.g., place order)
  2. Order Service saves data in its own database
  3. Order Service calls Payment Service via API
  4. Payment Service stores data in its database
  5. Services communicate via APIs/events — not direct DB access
