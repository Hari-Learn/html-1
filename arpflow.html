ARP Packet & Frame Flow

Learn what happens before ARP is called, how MAC tables and ARP caches are used, and the step-by-step packet & frame flow.

MAC Table (Switch)

Stores mappings: MAC Address → Switch Port

  • Learns source MAC addresses automatically
  • Used to forward frames efficiently
  • Destination MAC unknown → broadcast frame

ARP Cache (Host)

Stores mappings: IP Address → MAC Address

  • Checked before sending packets
  • If entry exists → ARP not required
  • If missing → ARP Request is triggered

Host A

IP: 192.168.1.10

MAC: AA:AA:AA:AA:AA:AA

Switch

Uses MAC Learning Table

Host B

IP: 192.168.1.20

MAC: BB:BB:BB:BB:BB:BB

Step-by-Step Packet & Frame Flow

  1. Application generates data
    Host A wants to send data to IP 192.168.1.20.
  2. Host checks ARP Cache (Before ARP)
    Entry missing → ARP Request is needed.
  3. Packet creation (Layer 3)
    IP packet created with Source IP: 192.168.1.10 and Destination IP: 192.168.1.20
  4. ARP Request (Broadcast)
    Ethernet frame fields:
    • Source MAC: AA:AA:AA:AA:AA:AA
    • Destination MAC: FF:FF:FF:FF:FF:FF
  5. Switch receives frame
    Switch performs MAC learning:
    • Learns AA:AA:AA → Incoming port
    • Destination MAC unknown → Broadcasts frame
  6. Host B receives ARP Request
    Matches target IP → prepares ARP Reply.
  7. ARP Reply (Unicast)
    Ethernet frame:
    • Source MAC: BB:BB:BB:BB:BB:BB
    • Destination MAC: AA:AA:AA:AA:AA:AA
  8. Switch updates MAC Table
    Learns BB:BB:BB → Port of Host B.
  9. Host A updates ARP Cache
    IP 192.168.1.20 → BB:BB:BB:BB:BB:BB
  10. Normal data transfer
    Frames now sent as unicast using learned MAC addresses.

Summary

  • ARP Cache is checked before sending ARP
  • Switch learns MAC addresses automatically
  • ARP Request = Broadcast, ARP Reply = Unicast
  • Once MACs are known, communication becomes efficient
