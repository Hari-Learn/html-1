OpenVAS / GVM — Quick Overview

A high‑level explanation of what OpenVAS (now part of GVM) is, its components, and responsible uses.

What is OpenVAS / GVM?

OpenVAS originally referred to the open‑source vulnerability scanner project. Today it's commonly encountered as part of the Greenbone Vulnerability Management (GVM) ecosystem. GVM is a framework for performing vulnerability scanning and producing reports used to find weaknesses in systems so they can be remediated.

Key components (high level)

  • OpenVAS/GVMD (scanner/engine): the core scanning engine that runs checks against targets.
  • GVM Manager (gvmd): manages scan tasks, schedules, credentials and stores results.
  • Greenbone Security Assistant (GSA): a web interface for configuring and viewing scans & reports.
  • Network Vulnerability Tests (NVTs) / feeds: the signature/knowledge base used to check vulnerabilities.
  • Reporting: produces results with severity indicators (e.g., CVSS) and remediation hints.

Scan types (conceptual)

Scans are typically authenticated (use credentials to assess internal configuration) or unauthenticated (external checks like open ports and known service issues). Different scan profiles focus on speed, depth, or compliance checks.

  • Unauthenticated discovery: finds open ports and visible services.
  • Authenticated checks: probe configuration and missing patches when credentials are provided.
  • Policy/compliance scans: validate settings against baselines or standards.

What the scanner outputs

Typical outputs include a list of findings with severity (low / medium / high), vulnerability identifiers (e.g., CVE/CVSS), references, and high‑level remediation suggestions. Reports are intended to guide patching and configuration fixes.

Example severity labels: LOW MEDIUM HIGH

Legitimate uses

  • Internal security assessments by authorized teams.
  • Vulnerability management workflows (discover → prioritize → remediate → verify).
  • Compliance validation and periodic security scanning.

Legal & ethical note

Scanning systems without explicit authorization may be illegal and unethical. Always have written permission (for example, a signed scope of work or a penetration testing authorization) before running any scans against systems you do not own.

Practical tips (non-actionable)

Learning & documentation

Start with the official GVM / Greenbone documentation and community resources for installation and safe usage. Look for:

  • Overview docs & architecture diagrams
  • Feed differences (community vs enterprise)
  • How to read reports and map findings to remediation

Operational best practices

  • Run scans in a controlled maintenance window if doing intrusive checks.
  • Prioritize findings by risk (business impact + severity).
  • Keep your vulnerability feeds and scanner software updated.

What to avoid

  • Avoid scanning external targets you don't own or aren't authorized to test.
  • Don't treat scan results as flawless — validate critical findings manually.
