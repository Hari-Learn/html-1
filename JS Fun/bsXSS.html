XSS — Explain Like I'm 10

Simple, harmless examples + how to prevent it. For learning and authorized testing only.

What is XSS?

Imagine a website is like a diary page that shows messages. If someone can make the diary show a secret message that runs code (a tiny program) instead of plain text, that's XSS. The website is letting someone put code into others' pages.

Why it matters (simple)

  • Bad code can run in visitors' browsers.
  • It can trick users or change what they see.
  • It's often used by attackers to pretend they are the site or to steal information — so it's serious.

Types of XSS (short)

  • Stored — code is saved on the website (e.g., comment) and shows to everyone later.
  • Reflected — code is in a link and shows when someone clicks it.
  • DOM XSS — the page's own scripts put unsafe text into the page and run it.

Harmless demo snippets

These show that code runs without doing anything harmful. Use only on sites you are allowed to test.

  • Popup (visible proof): <script>alert('test')</script>
  • Console message: <script>console.log('xss')</script>
  • DOM marker: <script>document.body.setAttribute('data-xss','1')</script>

How attackers often inject

  • Typing code into a comment box that the site later displays.
  • Putting code in a URL parameter (the link shows it back).
  • Making the page's JavaScript take user input and insert it into the page without checking.

Simple tests (safe)

Quick checks that don't steal data:

  • Enter <script>console.log('x')</script> into a text field and see if it appears as code or escaped text.
  • Try an image tag with an error handler that logs: <img src=x onerror="console.log('err')">
  • Look for unexpected HTML being shown instead of plain text.

How to prevent XSS (easy)

  • Escape output: turn special characters into harmless text before showing them (e.g., < becomes &lt;).
  • Use safe APIs: don't insert raw HTML with innerHTML; prefer text-only methods.
  • Validate input: check that inputs look like you expect (but escaping output is the best defense).
  • Content Security Policy (CSP): a rule that helps browsers block many XSS attacks.

Developer checklist (one line each)

  • Escape user data where it’s shown.
  • Avoid eval() and unsafe HTML insertion.
  • Use CSP to restrict scripts.
  • Sanitize rich text editors on the server.

Testing safety & ethics

  • Always have permission to test a site.
  • Use non-destructive checks (console, DOM markers, Collaborator for blind tests).
  • Don't exfiltrate real user data in tests.

Reporting an XSS

  • Show the exact input used and where you put it (form/URL).
  • Show the harmless proof (console message, DOM marker or Collaborator callback).
  • Explain impact: what an attacker could do if exploited.

Extra: Blind XSS

When the code runs somewhere you can't see (server-side logs, admin panels)

  • Use an OAST/Collaborator to get a callback proving execution.
  • Do not send real data in the payload; only proof tokens/adapters set up for testing.
