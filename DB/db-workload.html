Database Selection by Scale & Workload

Understand how database choice changes based on traffic scale and workload patterns. Focus on system behavior instead of just tools.

📈 3. Based on Scale & Traffic

🟢

Low / Medium Traffic Systems

Best for applications with predictable and limited user load.

What

Single-node or simple horizontally unscaled databases.

Why

Lower cost and easier to manage infrastructure.

How

Runs on a single server or basic replication setup.

Tools

MySQL, SQLite, PostgreSQL

🌍

High Traffic / Global Scale

Designed for distributed systems handling massive concurrent users.

What

Distributed databases spread across multiple regions.

Why

Ensures availability, fault tolerance, and scalability.

How

Data is partitioned (sharding) and replicated globally.

Tools

Cassandra, CockroachDB, DynamoDB

⚙️ 4. Based on Workload Pattern

📊

Read-Heavy Systems (OLAP)

Optimized for analytics, dashboards, and reporting workloads.

What

Systems that perform heavy aggregations and queries.

Why

Fast insights from large datasets.

How

Columnar storage and pre-aggregated data models.

Tools

Snowflake, BigQuery, Amazon Redshift

✍️

Write-Heavy Systems (OLTP)

Designed for frequent inserts, updates, and transactional systems.

What

Systems optimized for fast row-level transactions.

Why

Ensures consistency during frequent updates.

How

Row-based storage with indexing and locking mechanisms.

Tools

PostgreSQL, MySQL, SQL Server
